# TanStack Query: 페이지네이션 구현

## 구현 방식

React Query의 페이지네이션은 **컴포넌트 state로 현재 페이지를 직접 관리**하는 방식.  
(무한 스크롤은 React Query가 페이지를 관리 → 다음 섹션에서 다룸)

---

## 핵심 원리

페이지가 바뀔 때마다 **쿼리 키에 currentPage를 포함**해서 React Query가 새 쿼리로 인식하게 함.

```tsx
const [currentPage, setCurrentPage] = useState(1);

const { data, isLoading, isError, error } = useQuery({
  queryKey: ["posts", currentPage], // currentPage가 바뀌면 새 쿼리 실행
  queryFn: () => fetchPosts(currentPage),
  staleTime: 2000,
});
```

→ `currentPage`가 바뀌면 React Query가 키 변경을 감지하고 자동으로 새 데이터를 fetch

---

## 버튼 구현

```tsx
const maxPostPage = 10; // 전체 페이지 수

<button
  disabled={currentPage <= 1}
  onClick={() => setCurrentPage((prev) => prev - 1)}
>
  Previous page
</button>

<span>Page {currentPage}</span>

<button
  disabled={currentPage >= maxPostPage}
  onClick={() => setCurrentPage((prev) => prev + 1)}
>
  Next page
</button>
```

> 실제 서비스에서는 `maxPostPage`를 하드코딩하지 않고,  
> API 응답 데이터에 포함된 `hasNextPage` 등의 값으로 판단함

## 페이지 이동 시 동작 흐름

### 처음 방문하는 페이지 (캐시 없음)

```
4페이지 첫 방문
→ 캐시 없음
→ isLoading = true → 로딩 스피너 표시
→ fetch 완료 후 데이터 표시
```

### 이미 방문한 페이지로 돌아갈 때 (캐시 있음)

```
3페이지 → 2페이지 → 1페이지 (이미 방문한 페이지)
→ 캐시 있음
→ isLoading = false → 로딩 스피너 없이 즉시 표시
→ stale 상태라면 백그라운드에서 리패치 (네트워크 요청 O)
```

---

## 로딩은 없는데 네트워크 요청은 가는 이유

**stale-while-revalidate 패턴** 때문.

| 상태              | 화면                  |  네트워크 요청  |
| ----------------- | --------------------- | :-------------: |
| 캐시 없음         | 로딩 스피너           |       ✅        |
| 캐시 있음 + fresh | 캐시 데이터 즉시 표시 |       ❌        |
| 캐시 있음 + stale | 캐시 데이터 즉시 표시 | ✅ (백그라운드) |

→ stale 상태면 캐시 데이터를 **즉시 보여주면서** 뒤에서 **몰래 최신 데이터로 교체**

```
staleTime: 2000 (2초) 설정 시

1페이지 방문 → 2초 후 stale
다른 페이지 갔다가 1페이지 복귀 (2초 경과)
→ 캐시 데이터 즉시 표시 (isLoading = false, 로딩 없음)
→ 백그라운드 리패치 (네트워크 탭에 요청 찍힘)
```

---

## 문제점 & 다음 단계

현재 구현의 UX 문제: **처음 방문하는 페이지는 매번 로딩 스피너가 표시됨**

→ 해결책: **Prefetching** — 다음 페이지를 미리 fetch해서 캐시에 저장  
→ 다음 강의에서 다룸
