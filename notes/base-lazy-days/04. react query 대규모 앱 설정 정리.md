````markdown
# React Query 대규모 앱 설정 정리

## 1. QueryClient 별도 파일로 분리

```ts
// src/react-query/queryClient.ts
const queryClient = new QueryClient({
  queryCache: new QueryCache({
    onError: (error) => {
      // 에러 처리 중앙화
    },
  }),
});
```
````

컴포넌트(App.tsx)에 직접 쓰지 않고 별도 파일로 분리
→ 기본 옵션, 에러 콜백 등을 한 곳에서 관리 가능

---

## 2. 커스텀 훅으로 추상화

```ts
// useTreatments.ts
export function useTreatments() {
  const { data = [] } = useQuery({
    queryKey: [queryKeys.treatments],
    queryFn: getTreatments,
  });
  return data;
}
```

**장점**

- 같은 데이터를 여러 컴포넌트에서 쓸 때 편함
- queryKey, queryFn을 각 컴포넌트마다 신경 쓸 필요 없음

**예시 - treatments 데이터를 두 곳에서 사용**

```
useTreatments()
    ├── Treatments 컴포넌트 → 시술 목록 표시
    └── Staff 컴포넌트 → 필터링용 라디오 버튼
```

---

## 3. 로딩 중앙화

```ts
// App.tsx
const isFetching = useIsFetching()
// 현재 fetching 중인 쿼리 수 반환
// 0 → 로딩 없음, 1 이상 → 로딩 중

<Loading display={isFetching ? 'inherit' : 'none'} />
// 항상 DOM에 존재, display로 show/hide만 토글
```

- 각 컴포넌트마다 로딩 처리 불필요
- App 컴포넌트에서 전역으로 스피너 관리

---

## 4. 에러 처리 중앙화

```ts
const queryClient = new QueryClient({
  queryCache: new QueryCache({
    onError: (error) => {
      toast({
        title: "에러가 발생했습니다",
        description: error.message,
        status: "error",
      });
    },
  }),
});
```

- 각 컴포넌트마다 `isError` 처리 불필요
- 모든 useQuery 에러를 한 곳에서 toast로 표시

---

## 정리

|              | 기존 방식                  | 이 프로젝트 방식                 |
| ------------ | -------------------------- | -------------------------------- |
| QueryClient  | App.tsx에 직접             | 별도 파일 분리                   |
| 데이터 fetch | 컴포넌트에서 직접 useQuery | 커스텀 훅으로 추상화             |
| 로딩 처리    | 각 컴포넌트에서 isLoading  | App에서 useIsFetching으로 중앙화 |
| 에러 처리    | 각 컴포넌트에서 isError    | QueryCache onError로 중앙화      |

