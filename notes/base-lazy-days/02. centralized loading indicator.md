# 중앙 집중식 로딩 인디케이터 적용

## 핵심 요약

- 각 컴포넌트마다 개별 로딩 상태를 관리하지 않고, **앱 전체에서 공통 로딩 스피너**를 표시하는 방식으로 변경한다.
- 이를 위해 React Query의 **`useIsFetching` 훅**을 사용한다.
- `useIsFetching`은 현재 **fetch 중인 쿼리 개수**를 반환한다.
- 반환값이 **0보다 크면 로딩 스피너를 표시**하고, 0이면 숨긴다.
- 이 방식은 규모가 큰 앱에서 로딩 상태를 훨씬 일관되게 관리할 수 있게 해준다.

---

## 개요

이번 강의에서는 각 컴포넌트별 로딩 인디케이터 대신, **중앙 집중식 로딩 인디케이터**를 사용하는 방식으로 앱을 개선한다.

기존 소규모 앱에서는 `useQuery`의 반환값에서 `isFetching`을 구조 분해하여 각 컴포넌트 단위로 로딩 상태를 처리했다.
하지만 앱 규모가 커질수록, **앱 전체 어디에서든 데이터 요청이 발생하면 공통 스피너를 띄우는 방식**이 더 적합하다.

---

## `isLoading`과 `isFetching`의 차이

React Query에서 두 값은 비슷해 보이지만 의미가 다르다.

| 상태값           | 의미                                            |
| ---------------- | ----------------------------------------------- |
| **`isFetching`** | 현재 데이터를 가져오는(fetching) 중인지 여부    |
| **`isLoading`**  | fetching 중이면서 **캐시된 데이터도 없는 상태** |

### 정리

- **`isFetching`**이 더 큰 범주의 개념이다.
- **`isLoading`**은 그중에서도 **초기 로딩 상태**에 해당한다.

즉,

```txt
isLoading = isFetching + 캐시된 데이터 없음
```

---

## 왜 중앙 집중식 로딩이 필요한가

규모가 큰 앱에서는 여러 컴포넌트가 동시에 데이터를 요청할 수 있다.
이 경우 각 컴포넌트가 따로 로딩 UI를 가지면 다음과 같은 문제가 생길 수 있다.

- 로딩 표시 방식이 제각각이 된다.
- 중복된 로직이 많아진다.
- 사용자 입장에서 앱 전체 상태를 파악하기 어렵다.

그래서 **앱 루트 레벨에서 공통 로딩 스피너를 제어**하는 구조가 더 효율적이다.

---

## `useIsFetching` 훅의 역할

`useIsFetching`은 React Query에서 제공하는 훅으로,
현재 **fetch 상태인 모든 쿼리의 개수**를 반환한다.

### 특징

- 반환값은 **불리언이 아니라 정수(integer)** 이다.
- 예를 들어:
  - `0` → 현재 fetch 중인 쿼리가 없음
  - `1` 이상 → 하나 이상의 쿼리가 fetch 중임

즉, 이 값을 기반으로 스피너 표시 여부를 결정할 수 있다.

---

## 적용 방식

기존에는 로딩 컴포넌트가 항상 숨겨져 있었고, 예를 들어 `false` 같은 고정값으로 제어되고 있었다.

이제는 `useIsFetching`을 불러와 그 결과값으로 표시 여부를 제어한다.

### 예시 코드

```tsx
import { useIsFetching } from "@tanstack/react-query";

function Loading() {
  const isFetching = useIsFetching();

  return <Spinner display={isFetching ? "inherit" : "none"} />;
}
```

---

## 동작 원리

`useIsFetching()`이 반환한 값이 0보다 크면 truthy로 평가된다.

### 흐름

1. 어떤 쿼리든 fetch를 시작한다.
2. `useIsFetching()` 값이 증가한다.
3. 로딩 스피너가 표시된다.
4. 모든 fetch가 끝난다.
5. `useIsFetching()` 값이 0이 된다.
6. 로딩 스피너가 숨겨진다.

---

## 앱 구조 내 위치

로딩 컴포넌트는 **앱 전체에서 공통으로 보이는 위치**에 배치된다.

예를 들어 `App.tsx`에서 다음과 같이 구성할 수 있다.

```tsx
<Loading />
<NavBar />
<Routes />
<ReactQueryDevtools />
```

이렇게 하면 앱 어디에서 쿼리가 발생하든, `Loading` 컴포넌트가 이를 감지해 공통 스피너를 보여줄 수 있다.

---

## 실습 중 확인한 동작

실제로 화면 전환 후 로딩 스피너가 나타났지만,
데이터 요청이 너무 빨라서 거의 눈에 띄지 않았다.

그래서 브라우저 DevTools에서 네트워크 속도를 **Slow 3G**로 낮춘 뒤 다시 확인했다.

그 결과, 스피너가 더 명확하게 표시되는 것을 확인할 수 있었다.

---

## 창 포커스 시 다시 로딩되는 이유

강의 중 창을 다시 클릭했을 때도 스피너가 표시되었다.
이유는 React Query의 기본 설정 중 하나인 **window refocus 시 refetch** 때문이다.

즉,

- 브라우저 창이 다시 포커스를 얻으면
- React Query가 자동으로 데이터를 다시 요청하고
- 그에 따라 `useIsFetching()` 값이 증가하며
- 로딩 스피너가 표시된다.

이 부분은 이후 **Refetch 설정**에서 더 자세히 다룬다.

---

## 핵심 정리

### 이전 방식

- 각 커스텀 훅 또는 각 컴포넌트에서 `isFetching`을 따로 사용

### 현재 방식

- 앱 전역에서 `useIsFetching` 하나로 로딩 상태를 통합 관리

### 장점

- 로딩 UI 일관성 확보
- 중복 코드 감소
- 대규모 앱에 적합
- 유지보수 편리

---

## Takeaway

- **`useIsFetching`은 현재 fetch 중인 쿼리 개수를 반환하는 전역 상태용 훅**이다.
- **개별 컴포넌트 로딩 대신 앱 전체 공통 로딩 스피너 구현**에 적합하다.
- `isLoading`은 초기 로딩 상태이고, `isFetching`은 더 넓은 개념이다.
- React Query는 기본적으로 **창 포커스 시 refetch**를 수행하므로, 이때도 스피너가 나타날 수 있다.
- 큰 앱일수록 **중앙 집중식 로딩 처리 방식**이 더 실용적이다.
