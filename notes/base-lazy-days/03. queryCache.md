# Query Cache 전역 `onError` 콜백으로 에러 토스트 표시

## 핵심 요약

- React Query는 `useIsFetching`처럼 전역 에러를 읽는 **`useError` 훅** 대신, **Query Cache의 `onError` 콜백**을 제공한다.
- 이 콜백을 `QueryClient` 생성 시 설정하면, 어떤 `useQuery`에서 에러가 발생하든 **공통 토스트 메시지**를 띄울 수 있다.
- 에러 메시지는 보통 `error.message`를 사용해 표시한다.
- Chakra UI의 `toast.isActive(id)`를 활용하면 **중복 토스트 노출을 방지**할 수 있다.
- React Query는 기본적으로 **실패 시 3회 재시도**한 뒤 최종 에러 상태가 되며, 그 후 토스트가 보인다.

---

## 개요

이번 강의에서는 React Query의 **Query Cache 전역 에러 핸들링**을 설정하여,
어떤 `useQuery` 호출에서 에러가 발생하더라도 **Chakra UI Toast**로 사용자에게 에러를 알려주는 구조를 만든다.

즉, 각 쿼리마다 개별적으로 에러 UI를 붙이기보다,
**앱 전체에서 공통 에러 처리 방식**을 적용하는 것이다.

---

## 왜 `useError` 훅이 아니라 `onError` 콜백을 쓰는가

처음에는 `useIsFetching`처럼 전역 에러 상태를 알려주는 **`useError` 훅**이 있으면 좋겠다고 생각할 수 있다.
하지만 에러는 로딩 상태처럼 단순한 숫자로 표현하기 어렵다.

### 이유

- 로딩 상태는 “현재 몇 개의 쿼리가 fetch 중인가”처럼 **숫자 하나**로 표현 가능하다.
- 반면 에러는:
  - 어떤 쿼리에서 발생했는지
  - 어떤 메시지인지
  - 여러 에러가 동시에 발생했는지
    등을 함께 다뤄야 한다.

즉, 에러는 `useIsFetching`처럼 단순하고 깔끔하게 전역 훅으로 처리하기 어렵다.

그래서 React Query는 전역 훅 대신,
**Query Cache 수준의 `onError` 콜백**을 제공한다.

---

## 전역 에러 처리 방식

`QueryClient`를 생성할 때 `queryCache`를 추가하고,
그 안에 `onError` 콜백을 설정하면 된다.

### 흐름

1. 어떤 `useQuery`가 에러를 던진다.
2. Query Cache의 `onError`가 호출된다.
3. 에러 객체가 콜백 인자로 전달된다.
4. `error.message`를 꺼내 사용자용 토스트를 표시한다.

---

## 구현 코드

```ts
import { QueryCache, QueryClient } from "@tanstack/react-query";
import { toast } from "@/components/app/toast";

function errorHandler(errorMsg: string) {
  // https://chakra-ui.com/docs/components/toast#preventing-duplicate-toast
  // one message per page load, not one message per query
  // the user doesn't care that there were three failed queries on the staff page
  //    (staff, treatments, user)
  const id = "react-query-toast";

  if (!toast.isActive(id)) {
    const action = "fetch";
    const title = `could not ${action} data: ${
      errorMsg ?? "error connecting to server"
    }`;
    toast({ id, title, status: "error", variant: "subtle", isClosable: true });
  }
}

export const queryCleint = new QueryClient({
  queryCache: new QueryCache({
    onError: (error) => errorHandler(error.message),
  }),
});
```

---

## 코드 구조 설명

## 1. `errorHandler`

에러 토스트를 실제로 띄우는 함수다.

### 역할

- 에러 메시지를 받아 제목 문자열 생성
- 중복 토스트 방지
- Chakra UI Toast 표시

### 핵심 포인트

- `toast.isActive(id)`를 사용해 같은 ID의 토스트가 이미 떠 있으면 새로 띄우지 않는다.
- 같은 페이지에서 여러 쿼리가 동시에 실패해도 사용자에게 **토스트가 여러 개 겹쳐서 보이지 않게** 한다.

---

## 2. 토스트 ID로 중복 방지

```ts
const id = "react-query-toast";
```

이 ID는 토스트를 유일하게 식별하는 값이다.

### 의도

예를 들어 staff 페이지에서 아래 3개 쿼리가 모두 실패할 수 있다.

- staff
- treatments
- user

이때 사용자 입장에서는
“쿼리 3개가 각각 실패했다”는 사실보다
“현재 데이터를 불러오지 못했다”는 사실이 더 중요하다.

그래서 **페이지 로드당 하나의 에러 메시지만 보여주도록 설계**한다.

---

## 3. `action` 변수

```ts
const action = "fetch";
```

현재는 쿼리 에러만 처리하므로 `fetch`로 고정되어 있다.

토스트 제목은 다음처럼 만들어진다.

```ts
could not fetch data: ...
```

강의에서는 이후 이 함수를 **mutation 에러에도 재사용**할 계획이므로,
나중에는 `fetch`, `update`, `delete` 같은 식으로 바꿔 사용할 수 있게 설계한 것이다.

---

## 4. 에러 메시지 처리

```ts
const title = `could not ${action} data: ${
  errorMsg ?? "error connecting to server"
}`;
```

### 의미

- `errorMsg`가 있으면 그 값을 사용
- 없으면 기본 메시지인 `"error connecting to server"` 사용

즉, 서버 응답이 명확하지 않더라도 최소한 사용자에게
**서버 연결 문제**가 있었다는 안내는 가능하다.

---

## 5. Chakra UI Toast 옵션

```ts
toast({ id, title, status: "error", variant: "subtle", isClosable: true });
```

### 주요 옵션

| 옵션                    | 의미                       |
| ----------------------- | -------------------------- |
| **`id`**                | 중복 토스트 식별용 ID      |
| **`title`**             | 사용자에게 보여줄 제목     |
| **`status: "error"`**   | 에러 스타일 토스트         |
| **`variant: "subtle"`** | Chakra UI의 표시 스타일    |
| **`isClosable: true`**  | 사용자가 직접 닫을 수 있음 |

이 옵션들은 **Chakra UI 설정**이며, React Query 자체 기능은 아니다.

---

## Query Client에 연결하는 방법

에러 핸들러를 만든 뒤에는 `QueryClient` 생성 시 `queryCache` 옵션을 추가하면 된다.

```ts
export const queryCleint = new QueryClient({
  queryCache: new QueryCache({
    onError: (error) => errorHandler(error.message),
  }),
});
```

### 동작 방식

- `new QueryCache({...})` 생성
- `onError` 속성에 콜백 등록
- 쿼리 실패 시 `error` 객체 전달
- `error.message`를 추출하여 `errorHandler` 실행

---

## 실습에서 확인한 동작

강의에서는 서버를 일부러 중지한 뒤,
`Treatments` 컴포넌트가 서버 데이터를 요청하도록 해 에러를 발생시켰다.

### 결과

1. 페이지가 먼저 로딩 상태에 들어간다.
2. React Query가 기본 재시도 로직에 따라 요청을 다시 시도한다.
3. 기본적으로 **3번 재시도**한다.
4. 최종적으로 실패하면 에러 토스트가 표시된다.
5. 사용자에게 **"could not fetch data"** 메시지가 보인다.

---

## React Query 기본 재시도 동작

강의에서 중요한 포인트 중 하나는,
에러가 즉시 발생하는 것이 아니라 **재시도 후 최종 실패 처리**가 된다는 점이다.

### 기본값

- React Query는 기본적으로 **3회 재시도**한다.

즉, 서버가 잠시 불안정한 경우 자동 복구를 시도한 뒤,
그래도 실패하면 최종적으로 에러를 사용자에게 보여준다.

---

## 이 방식의 장점

### 1. 전역 에러 처리 가능

- 각 `useQuery`마다 같은 에러 처리 코드를 반복할 필요가 없다.

### 2. 사용자 경험 개선

- 화면 곳곳에서 에러가 따로 뜨는 대신, 공통 방식으로 정리된 에러 안내를 제공한다.

### 3. 중복 메시지 방지

- 여러 쿼리가 동시에 실패해도 토스트가 난잡하게 여러 개 뜨지 않는다.

### 4. 확장성 있음

- 이후 mutation 에러까지 같은 핸들러 구조로 확장 가능하다.

## 주의할 점

## 1. `queryCleint` 오타

코드에는 다음처럼 작성되어 있다.

```ts
export const queryCleint = new QueryClient({
```

여기서 `queryCleint`는 일반적으로 `queryClient`가 맞다.

### 권장

```ts
export const queryClient = new QueryClient({
  queryCache: new QueryCache({
    onError: (error) => errorHandler(error.message),
  }),
});
```

---

## 2. `error.message` 전제

현재 코드는 `onError: (error) => errorHandler(error.message)`로 작성되어 있다.

이 방식은 **`error`가 JavaScript `Error` 객체라고 가정**한다.
실무에서는 커스텀 에러 객체나 서버 에러 응답 형식이 다를 수 있으므로, 필요하면 방어적으로 처리하는 편이 안전하다.

예시:

```ts
onError: (error) => {
  const message = error instanceof Error ? error.message : "unknown error";
  errorHandler(message);
};
```

## 정리 포인트

| 항목             | 내용                                 |
| ---------------- | ------------------------------------ |
| 에러 처리 위치   | `QueryClient`의 `queryCache.onError` |
| 에러 표시 방식   | Chakra UI Toast                      |
| 중복 방지 방법   | `toast.isActive(id)`                 |
| 에러 메시지 소스 | `error.message`                      |
| 기본 재시도 횟수 | 3회                                  |
| 목적             | 앱 전역 쿼리 에러를 일관되게 처리    |

---

## Takeaway

- React Query는 전역 에러 훅 대신 **`QueryCache`의 `onError` 콜백**을 제공한다.
- 이를 이용하면 어떤 `useQuery`가 실패하더라도 **공통 토스트 메시지**를 띄울 수 있다.
- `toast.isActive(id)`를 사용하면 **중복 에러 메시지 노출을 막을 수 있다**.
- React Query는 기본적으로 **3번 재시도 후 최종 실패**를 처리한다.
- 이 구조는 작은 앱보다 **규모가 큰 앱에서 특히 유용한 전역 에러 처리 패턴**이다.
