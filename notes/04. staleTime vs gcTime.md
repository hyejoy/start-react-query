# TanStack Query: `staleTime` vs `gcTime`

## stale 상태란?

- 데이터가 **만료**되어 리패치가 필요한 상태
- stale이어도 **캐시에서 제거되지 않음** — 화면에 계속 표시 가능
- 캐시에 남아있되, 서버에서 **최신 데이터로 재검증**이 필요하다는 신호

> 💡 SWR(Vercel)의 개념과 동일 — **S**tale **W**hile **R**evalidating  
> stale 데이터를 보여주면서, 백그라운드에서 새 데이터를 fetch

---

## staleTime

> "얼마나 오래 데이터를 신선하다고 볼 것인가?"

- 데이터를 **fresh 상태로 유지하는 시간** (밀리초 단위)
- 이 시간이 지나면 데이터가 `stale` 상태로 전환
- **stale 상태일 때만** 리패치 트리거가 동작함

### 기본값: `0ms`
데이터를 항상 오래된 것으로 간주 → 매번 최신 데이터를 서버에서 가져옴  
→ 실수로 오래된 데이터를 보여줄 가능성 최소화

```tsx
const { data } = useQuery({
  queryKey: ["posts"],
  queryFn: fetchPosts,
  staleTime: 2000, // 2초 동안 fresh 유지
});
```

### 자동 리패치 트리거 (데이터가 stale일 때만 동작)
- 쿼리를 사용하는 컴포넌트가 **리마운트**될 때
- 브라우저 **윈도우가 다시 포커스**될 때

---

## gcTime (Garbage Collection Time)

> "사용하지 않는 캐시 데이터를 얼마나 보관할 것인가?"

- 데이터를 **사용하는 컴포넌트가 없을 때** GC 타이머 시작
- 타이머가 만료되면 캐시에서 **완전히 삭제**
- 삭제된 데이터는 리액트 쿼리가 전혀 모르는 상태가 됨

### 기본값: `5분`

```tsx
const { data } = useQuery({
  queryKey: ["posts"],
  queryFn: fetchPosts,
  gcTime: 1000 * 60 * 5, // 5분 (기본값)
});
```

> ⚠️ GC 타이머는 데이터가 **화면에 표시되는 동안에는 작동하지 않음**  
> 컴포넌트가 언마운트되는 순간부터 카운트 시작

---

## staleTime vs gcTime 비교

| | `staleTime` | `gcTime` |
|---|---|---|
| 역할 | 리패치 필요 여부 결정 | 캐시 보관 기간 결정 |
| 기본값 | `0ms` | `5분` |
| 만료 시 | 데이터가 `stale` 상태로 전환 | 캐시에서 완전 삭제 |
| 데이터 표시 | 만료 후에도 캐시에 남아 표시 가능 | 만료 후 표시 불가 |

---

## 상태 조합별 동작

| 데이터 상태 | 캐시 여부 | 동작 |
|---|---|---|
| **fresh** | 캐시 있음 | 캐시 데이터 표시, 리패치 ❌ |
| **stale** | 캐시 있음 | 캐시 데이터 표시하면서 백그라운드 리패치 ✅ |
| — | 캐시 없음 (GC 만료 or 최초) | 서버 응답 올 때까지 표시할 데이터 없음 |
