# useInfiniteQuery

## useQuery vs useInfiniteQuery 데이터 구조 차이

```ts
// useQuery
data = [...] // 단순 데이터 배열

// useInfiniteQuery
data = {
  pages: [page1, page2, page3, ...], // 각 페이지 데이터 배열
  pageParams: [param1, param2, ...]  // 각 페이지의 파라미터 (거의 안 씀)
}
```

> `pages` 배열의 각 요소 = API 응답 객체 하나 (results, next, previous, count 전부 포함)

---

## 기본 문법

```ts
const { data, fetchNextPage, hasNextPage } = useInfiniteQuery({
  queryKey: ["sw-people"],
  queryFn: ({ pageParam = initialUrl }) => fetchUrl(pageParam),
  getNextPageParam: (lastPage) => lastPage.next || undefined,
});
```

---

## 각 옵션 설명

### queryFn

```js
({ pageParam = initialUrl }) => fetchUrl(pageParam);
```

- `pageParam`으로 fetch 실행
- 처음엔 기본값(`initialUrl`)으로 실행
- 이후엔 `getNextPageParam`이 반환한 값이 `pageParam`으로 들어옴

### getNextPageParam

```js
(lastPage) => lastPage.next || undefined;
```

**lastPage가 뭐냐?**

```js
// API에서 받은 응답 객체 그대로
lastPage = {
  count: 82,
  next: "https://swapi.dev/api/people/?page=2",  // 다음 페이지 URL
  previous: null,
  results: [{ name: "Luke", ... }, ...]  // 실제 데이터
}
```

**반환값은 어디로 가냐?**

```
getNextPageParam → "https://swapi.dev/api/people/?page=2" 반환
        ↓
pageParam 업데이트
        ↓
사용자 스크롤 → fetchNextPage() 호출
        ↓
queryFn({ pageParam: "https://swapi.dev/api/people/?page=2" })
        ↓
2페이지 fetch!
```

> getNextPageParam의 반환값 = 다음 번 queryFn에 들어갈 pageParam

**undefined 반환하면?**

```js
lastPage.next = null  // 마지막 페이지
→ undefined 반환
→ hasNextPage = false → 더 이상 fetch 안 함
```

---

## 반환 객체 주요 속성

| 속성                 | 설명                                         |
| -------------------- | -------------------------------------------- |
| `fetchNextPage`      | 다음 페이지 fetch 함수 (스크롤 감지 시 호출) |
| `hasNextPage`        | 다음 페이지 존재 여부                        |
| `isFetchingNextPage` | 다음 페이지 fetch 중 여부                    |

### hasNextPage 판단 기준

| getNextPageParam 반환값 | hasNextPage |
| ----------------------- | ----------- |
| URL 또는 값 있음        | `true`      |
| `undefined`             | `false`     |

---

## 전체 흐름

```
컴포넌트 마운트
    ↓
queryFn(pageParam = initialUrl) → 1페이지 fetch
    ↓
data.pages[0] = { results: [...], next: "...?page=2" }
    ↓
getNextPageParam(data.pages[0]) → "...?page=2" 반환 → pageParam 업데이트
    ↓
hasNextPage = true
    ↓
사용자 스크롤 → fetchNextPage()
    ↓
queryFn(pageParam = "...?page=2") → 2페이지 fetch
    ↓
data.pages[1] = { results: [...], next: "...?page=3" }
    ↓
getNextPageParam(data.pages[1]) → "...?page=3" 반환
    ↓
... 반복 ...
    ↓
getNextPageParam(lastPage) → undefined (next = null)
    ↓
hasNextPage = false → 종료!
```

---

## API 구조에 따라 달라지는 getNextPageParam

```js
// Star Wars API → next URL 있으니까
getNextPageParam: (lastPage) => lastPage.next || undefined;

// 페이지 번호 기반 API
getNextPageParam: (lastPage, allPages) => allPages.length + 1;

// cursor 기반 API
getNextPageParam: (lastPage) => lastPage.cursor;
```

> API 응답 구조에 따라 "다음 fetch할 때 pageParam으로 뭘 쓸지" 반환하면 됨

# useInfiniteQuery - API 구조별 사용법

## 핵심 원칙

> API 응답 구조가 어떻든 **"다음 fetch할 때 pageParam으로 뭘 넘길지"** 만 결정할 수 있으면 사용 가능

---

## API 구조별 getNextPageParam 작성법

### 1. next URL 기반 (Star Wars API)

```js
// 응답
{ results: [...], next: "https://swapi.dev/api/people/?page=2", ... }

// 설정
queryFn: ({ pageParam = initialUrl }) => fetchUrl(pageParam)
getNextPageParam: (lastPage) => lastPage.next || undefined
```

### 2. 페이지 번호 기반

```js
// 응답 (그냥 배열)
[{ id: 1, name: "..." }, ...]

// 설정
queryFn: ({ pageParam = 1 }) => fetch(`/api/posts?page=${pageParam}`)
getNextPageParam: (lastPage, allPages) => allPages.length + 1
```

### 3. cursor 기반

```js
// 응답
{ data: [...], cursor: "abc123" }

// 설정
queryFn: ({ pageParam }) => fetch(`/api/posts?cursor=${pageParam}`)
getNextPageParam: (lastPage) => lastPage.cursor
```

---

## 정리

| API 타입         | pageParam   | getNextPageParam 반환값 |
| ---------------- | ----------- | ----------------------- |
| next URL 기반    | 첫 번째 URL | `lastPage.next`         |
| 페이지 번호 기반 | 1           | `allPages.length + 1`   |
| cursor 기반      | undefined   | `lastPage.cursor`       |

> 더 이상 데이터가 없으면 `undefined` 반환 → `hasNextPage = false`
