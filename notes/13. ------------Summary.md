# TanStack Query: 섹션 1~2 총정리

## 1. 설치 및 기본 세팅

```tsx
// QueryClient 생성 + QueryClientProvider로 감싸기
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";

const queryClient = new QueryClient();

function App() {
  return (
    <QueryClientProvider client={queryClient}>
      <App />
    </QueryClientProvider>
  );
}
```

- `QueryClientProvider` 내부의 모든 자식 컴포넌트에서 캐시와 훅 사용 가능

---

## 2. useQuery — 데이터 조회

```tsx
const { data, isLoading, isFetching, isError, error } = useQuery({
  queryKey: ["posts", currentPage], // 필수, 항상 배열
  queryFn: () => fetchPosts(currentPage), // 필수, Promise 반환 함수
  staleTime: 2000,
});
```

### 주요 반환 상태

| 상태         | 설명                                     |
| ------------ | ---------------------------------------- |
| `data`       | 쿼리 성공 시 반환 데이터                 |
| `isLoading`  | 캐시 없음 + 요청 진행 중 (로딩 스피너용) |
| `isFetching` | 요청 진행 중 (캐시 유무 무관)            |
| `isError`    | 요청 실패                                |
| `error`      | 에러 객체                                |

> 조기 반환엔 `isLoading` 사용 — `isFetching`은 캐시 있어도 화면을 막아버림

## 3. staleTime vs gcTime

|         | `staleTime`                     | `gcTime`           |
| ------- | ------------------------------- | ------------------ |
| 역할    | 리패치 필요 여부 결정           | 캐시 보관 기간     |
| 기본값  | `0ms` (항상 stale)              | `5분`              |
| 만료 시 | stale 상태 → 리패치 트리거 동작 | 캐시에서 완전 삭제 |

```
fresh 상태 → 리패치 트리거 와도 무시
stale 상태 → 트리거 발생 시 캐시 보여주면서 백그라운드 리패치
캐시 없음  → 서버 응답 올 때까지 로딩
```

### 자동 리패치 트리거 (stale일 때만 동작)

- 컴포넌트 리마운트
- 브라우저 윈도우 포커스
- 수동 `refetch()` 호출
- 지정 인터벌
- `invalidateQueries` 호출

## 4. queryKey = 의존성 배열

```tsx
// ❌ 모든 post가 같은 캐시 공유
queryKey: ["comments"];

// ✅ post마다 독립 캐시 슬롯
queryKey: ["comments", post.id];
```

> **쿼리 함수 안에서 사용하는 값은 반드시 queryKey에 포함**  
> key가 바뀌면 React Query가 새 쿼리로 인식 → 자동 리패치

## 5. 페이지네이션

```tsx
const [currentPage, setCurrentPage] = useState(1);

useQuery({
  queryKey: ["posts", currentPage], // 페이지 변경 시 새 쿼리
  queryFn: () => fetchPosts(currentPage),
});
```

- 컴포넌트 state로 현재 페이지 관리
- `currentPage`가 바뀌면 React Query가 자동으로 새 데이터 fetch
- 처음 방문 페이지 → 로딩 스피너 / 재방문 페이지 → 캐시 즉시 표시

## 6. Prefetching

```tsx
const queryClient = useQueryClient();

useEffect(() => {
  if (currentPage < maxPostPage) {
    queryClient.prefetchQuery({
      queryKey: ["posts", currentPage + 1],
      queryFn: () => fetchPosts(currentPage + 1),
      // staleTime 미설정 시 기본값 0 → 즉시 stale
    });
  }
}, [currentPage, queryClient]);
```

- 다음 페이지를 미리 캐시에 저장 → 이동 시 로딩 없음
- prefetch 데이터는 즉시 stale → 페이지 이동 시 캐시 표시 + 백그라운드 리패치

## 7. useMutation — 데이터 변경 (CUD)

```tsx
const deleteMutation = useMutation({
  mutationFn: (postId) => deletePost(postId),
  // queryKey 없음 — 캐시와 무관
});

// 실행
deleteMutation.mutate(post.id);
```

### useQuery vs useMutation

|                | `useQuery`                 | `useMutation`     |
| -------------- | -------------------------- | ----------------- |
| 용도           | 데이터 조회 (R)            | 데이터 변경 (CUD) |
| queryKey       | ✅ 필수                    | ❌ 불필요         |
| 캐시           | ✅ 있음                    | ❌ 없음           |
| 실패 시 재시도 | 기본 3회                   | 기본 0회          |
| 로딩 상태      | `isLoading` / `isFetching` | `isPending`       |

### mutation 상태값

```tsx
{
  mutationObj.isPending && <p>요청 중...</p>;
}
{
  mutationObj.isSuccess && <p>성공!</p>;
}
{
  mutationObj.isError && <p>에러: {mutationObj.error.toString()}</p>;
}
```

## 8. mutation reset

```tsx
// 다른 포스트 클릭 시 이전 mutation 상태 초기화
onClick={() => {
  deleteMutation.reset();
  updateMutation.reset();
  setSelectedPost(post);
}}
```

- mutation을 부모 컴포넌트에서 선언 → 상태 안정적으로 유지
- 자식 컴포넌트가 바뀔 때 `reset()`으로 수동 초기화 필수
