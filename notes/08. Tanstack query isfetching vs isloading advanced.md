# TanStack Query: `isFetching` vs `isLoading` 실전 비교

## 핵심 차이

```
isFetching = 서버로 네트워크 요청이 진행 중인 순간 (캐시 유무 무관)
isLoading  = 네트워크 요청 중 + 캐시 데이터도 없는 상태
```

> `isLoading`은 `isFetching`의 부분집합  
> `isLoading === true`이면 `isFetching === true`는 항상 성립  
> 반대는 성립 안 함

---

## 조기 반환에 뭘 써야 할까?

### `isFetching`으로 조기 반환할 경우

```tsx
if (isFetching) return <h3>Fetching...</h3>; // ❌ 권장하지 않음
```

```
캐시 데이터 있음 + 백그라운드 리패치 중
→ isFetching = true
→ 캐시 데이터가 있는데도 화면을 막아버림
→ prefetch가 의미 없어짐
```

### `isLoading`으로 조기 반환할 경우

```tsx
if (isLoading) return <h3>Loading...</h3>; // ✅ 권장
```

```
캐시 데이터 있음 + 백그라운드 리패치 중
→ isLoading = false
→ 캐시 데이터 즉시 표시, 백그라운드에서 조용히 리패치
→ prefetch가 제대로 동작함
```

---

## Prefetch의 staleTime 주의점

`prefetchQuery`와 `useQuery`의 staleTime은 **별개로 설정**됨.

```tsx
// useQuery: staleTime 2초
const { data } = useQuery({
  queryKey: ["posts", currentPage],
  queryFn: () => fetchPosts(currentPage),
  staleTime: 2000,
});

// prefetchQuery: staleTime 별도 설정 안 하면 기본값 0
queryClient.prefetchQuery({
  queryKey: ["posts", nextPage],
  queryFn: () => fetchPosts(nextPage),
  // staleTime 미설정 → 기본값 0 → 즉시 stale
});
```

---

## Prefetch 후 페이지 이동 시 전체 흐름

```
1페이지에서 2페이지 데이터 prefetch
→ ["posts", 2] 캐시에 저장, 즉시 stale (staleTime: 0)

next page 클릭 → currentPage = 2
→ useQuery가 ["posts", 2] 캐시 확인
→ 캐시 있음 → 즉시 화면에 표시 (isLoading = false, 로딩 없음)
→ stale 상태 → 백그라운드에서 리패치 실행 (isFetching = true)
→ 응답 데이터 === 캐시 데이터 → 화면 변화 없음 (사용자는 모름)
→ 응답 데이터 !== 캐시 데이터 → 화면 업데이트
```

---

## 상태 조합 정리

| 상황                          | `isFetching` | `isLoading` | 화면             |
| ----------------------------- | :----------: | :---------: | ---------------- |
| 최초 로드 (캐시 없음)         |      ✅      |     ✅      | 로딩 스피너      |
| 캐시 있음 + 백그라운드 리패치 |      ✅      |     ❌      | 캐시 데이터 표시 |
| 캐시 있음 + fresh             |      ❌      |     ❌      | 캐시 데이터 표시 |

---

## 결론

- 조기 반환 조건은 **`isLoading`** 사용
- **`isFetching`** 은 로딩 스피너 대신 상단 인디케이터 등 보조 UI에 활용
- prefetch의 staleTime 기본값은 **0** → 캐시에 저장되자마자 stale
- 덕분에 다음 페이지 이동 시 **캐시 즉시 표시 + 백그라운드 리패치** 동작
